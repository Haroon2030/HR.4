# قاعدة بيانات واحدة (Neon)

النظام يقرأ **مصدراً واحداً فقط**: متغير `DATABASE_URL` في بيئة التشغيل (محلياً `.env`، على السيرفر Dokploy → Environment).

لا يوجد في الكود قاعدتان في آن واحد. إذا رأيت بيانات على الموقع ولا تراها في Neon، فالموقع ما زال يتصل بـ **Postgres داخلي** في Dokploy (`hr-hrpostgres-...`) وليس برابط Neon.

---

## الهدف

| البيئة | `DATABASE_URL` |
|--------|----------------|
| محلي (تطوير) | Neon (pooler) — نفس الرابط |
| إنتاج (Dokploy) | **نفس** رابط Neon — لا Postgres منفصل للتطبيق |

---

## 1) ضبط Dokploy (إلزامي)

1. افتح التطبيق **HR** في Dokploy → **Environment**.
2. عيّن `DATABASE_URL` إلى رابط Neon (**pooler**):

   ```text
   postgresql://USER:PASSWORD@ep-....pooler....neon.tech/hr?sslmode=require
   ```

   - إن كان `@` داخل كلمة المرور: رمّزه `%40`.
   - `channel_binding=require` اختياري؛ يمكن حذفه إن فشل الاتصال من الحاوية.

3. احذف أو **لا تربط** متغيرات قديمة مثل:
   - `POSTGRES_HOST=hr-hrpostgres-...`
   - أي `DATABASE_URL` يشير إلى `5432` داخل شبكة Docker فقط.

4. **Redeploy** التطبيق (ليس فقط Restart) حتى تُحمَّل المتغيرات.

5. بعد التشغيل، من داخل الحاوية:

   ```bash
   cd /app
   python manage.py check_production_data --details
   ```

   يجب أن يظهر `host: ep-....neon.tech` وليس `hr-hrpostgres`.

---

## 2) نقل البيانات من Postgres القديم إلى Neon

إذا كان الموقع يعمل اليوم على قاعدة Dokploy الداخلية (~10 آلاف بصمة، موظفون، …) وNeon فارغة:

### أ) من نسخة احتياطية على R2 (إن وُجدت)

```bash
cd /app
# تأكد أن DATABASE_URL يشير إلى Neon قبل الاسترجاع
python manage.py restore_db --list
python manage.py restore_db --latest --confirm
python manage.py migrate --noinput
python manage.py check_production_data --details
```

### ب) نسخ مباشر (pg_dump → Neon)

على جهاز يصل للقاعدتين:

```bash
# 1) تصدير من القاعدة القديمة (Dokploy)
pg_dump "postgresql://postgres:OLD_PASS@hr-hrpostgres-...:5432/hr" \
  --no-owner --no-acl -Fc -f hr_old.dump

# 2) استيراد إلى Neon
pg_restore -d "postgresql://neondb_owner:PASS@ep-....neon.tech/hr?sslmode=require" \
  --no-owner --no-acl --clean --if-exists hr_old.dump
```

ثم غيّر `DATABASE_URL` في Dokploy إلى Neon وأعد النشر.

### ج) بدء من صفر على Neon

إذا لا تحتاج البيانات القديمة:

1. ضع `DATABASE_URL` = Neon.
2. Redeploy → `migrate` ينشئ الجداول.
3. أعد إدخال البيانات أو `import_initial_data` إن لزم.
4. أعد مزامنة أجهزة البصمة والوكيل.

---

## 3) إيقاف ازدواجية خدمة Postgres في Dokploy (اختياري)

بعد التأكد أن كل شيء يعمل على Neon:

- أوقف أو احذف خدمة **Postgres** المنفصلة في Dokploy إن لم تعد تستخدمها لشيء آخر.
- هذا يوفر موارداً ويمنع الخلط مستقبلاً.

---

## 4) المحلي (.env)

استخدم **نفس** `DATABASE_URL` لـ Neon في `backend/.env` أو جذر المشروع:

```bash
cd backend
python manage.py migrate
python manage.py check_production_data --details
```

---

## 5) الجداول المتعلقة بالبصمة على Neon

| الجدول | المحتوى |
|--------|---------|
| `attendance_attendancepunch` | سجلات الحضور |
| `attendance_biometricdevice` | الأجهزة |
| `attendance_biometricdeviceuser` | مستخدمون على الجهاز |
| `attendance_employeebiometricenrollment` | ربط موظف ↔ جهاز |

```sql
SELECT COUNT(*) FROM attendance_attendancepunch WHERE is_deleted = false;
```

---

## 6) أمان

- لا تضع `DATABASE_URL` الحقيقي في Git.
- إذا ظهر الرابط في محادثة أو لقطة شاشة: **غيّر كلمة مرور Neon** من [console.neon.tech](https://console.neon.tech) ثم حدّث Dokploy و`.env`.

---

## التحقق السريع

```bash
python manage.py check_production_data
python manage.py check_attendance_production --details
```

نجاح الاتصال + ظهور `neon.tech` في سطر الاتصال = قاعدة موحدة صحيحة.
