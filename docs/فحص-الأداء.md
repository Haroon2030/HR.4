# فحص أداء نظام الموارد البشرية

دليل عملي لقياس سرعة الموقع في المتصفح وللتحقق بعد نشر التحسينات.

---

## 1. إعداد فحص صحيح (Chrome / Edge)

1. افتح الموقع في **نافذة خاصة** (لتعطيل إضافات المتصفح).
2. اضغط `F12` → تبويب **Network**.
3. اختر الفلتر **All** (وليس Fetch/XHR فقط).
4. فعّل **Disable cache** أثناء أول فحص.
5. حدّث الصفحة بـ `Ctrl+Shift+R` (تحديث قوي).

### ما الذي تبحث عنه؟

| المؤشر | جيد | يحتاج تحسين |
|--------|-----|-------------|
| **TTFB** لصفحة HTML | أقل من 500ms | أكثر من 1s |
| **DOMContentLoaded** | أقل من 2s | أكثر من 4s |
| **Load** | أقل من 3s | أكثر من 6s |
| حجم **tailwind.min.css** (بعد تسجيل الدخول) | ~300KB مضغوط | بدون gzip |
| صفحة **تسجيل الدخول** | `login.css` صغير (~3KB) | تحميل tailwind كامل |

---

## 2. مسارات الاختبار

| الخطوة | الرابط | الهدف |
|--------|--------|--------|
| 1 | `/health/` | سرعة السيرفر بدون قوالب |
| 2 | `/auth/login/` | أول انطباع (CSS خفيف) |
| 3 | `/` بعد الدخول | لوحة التحكم + استعلامات DB |

---

## 3. تبويب Performance (اختياري)

1. `F12` → **Performance** → تسجيل أثناء فتح لوحة التحكم.
2. راجع **LCP** (أول محتوى مرئي) و**Long Tasks** (JavaScript ثقيل).

---

## 4. أخطاء شائعة تُظهر شاشة بيضاء

- فلتر **Fetch/XHR** يخفي طلب HTML.
- إضافة متصفح (ملفات مثل `content_main.js` / `default_config.content.json`).
- `USE_HTTPS=true` بينما الدخول عبر `http://IP:8082` — اضبط `.env` كما في `.env.example`.

---

## 5. فحص السيرفر (سطر أوامر)

```bash
curl -sI -H "Accept-Encoding: gzip" https://YOUR_HOST/static/css/login.css
curl -sI -H "Accept-Encoding: gzip" https://YOUR_HOST/health/
```

تأكد من وجود `Content-Encoding: gzip` للملفات الثابتة.

---

## 6. إعدادات الإنتاج الموصى بها

```env
REDIS_URL=redis://...
SIDEBAR_COUNTS_CACHE_TTL=45
WHITENOISE_MAX_AGE=31536000
COLLECTSTATIC_ON_START=false
```

- **Redis**: ذاكرة مشتركة بين workers Gunicorn.
- **COLLECTSTATIC_ON_START**: `false` بعد أول نشر (الملفات تُجمَّع عند بناء Docker).

---

## 7. بعد كل نشر

1. افتح `/auth/login/` — يجب تحميل `login.css` فقط (بدون tailwind).
2. سجّل الدخول — راقب عدد طلبات DB في السجلات (يجب أن تقل العدادات بفضل التخزين المؤقت).
3. كرر الفحص من شبكة الجوال (DevTools → Throttling → Fast 3G).

---

## 8. التحسينات المطبّقة في الكود

| التحسين | الملف / المكوّن |
|---------|------------------|
| CSS خفيف لصفحة الدخول | `static/css/login.css`, `base_public.html` |
| تخزين مؤقت لعدادات القائمة | `services/sidebar_counts.py` |
| إبطال العدادات عند التحديث | `signals.py` |
| تجميع static عند بناء Docker | `Dockerfile` |
| تخطي collectstatic عند التشغيل | `entrypoint.sh` + `COLLECTSTATIC_ON_START` |
| Cache-Control للملفات الثابتة | `WHITENOISE_MAX_AGE` في الإعدادات |

## 9. فحص البصمة وقاعدة البيانات (الإنتاج)

داخل حاوية Docker على السيرفر (**مهم:** مجلد العمل `/app` وليس `/`):

```bash
cd /app
python manage.py check_attendance_production
python manage.py check_attendance_production --details
python manage.py check_attendance_production --deploy   # يُشغَّل تلقائياً عند النشر
python manage.py check_attendance_production --json
```

أو سطر واحد:

```bash
cd /app && python manage.py check_attendance_production --details
```

إذا ظهر `can't open file manage.py` فأنت في المجلد الخاطئ — نفّذ `cd /app` أولاً.

**وضع `--deploy`:** يُنفَّذ في `docker/entrypoint.sh` وGitHub Actions بعد كل `migrate`. يوقف النشر فقط عند: فشل DB، جداول ناقصة، أو `ATTENDANCE_AGENT_API_KEY` غير مضبوط في الإنتاج. جدول بصمات فارغ = تحذير فقط.

يفحص: اتصال PostgreSQL، وجود الجداول الأربعة، مفتاح الوكيل، الأجهزة، حفظ السجلات في `attendance_attendancepunch`، وربط الموظفين.

| الجدول | الغرض |
|--------|--------|
| `attendance_biometricdevice` | أجهزة البصمة |
| `attendance_biometricdeviceuser` | مستخدمو الجهاز (من الوكيل) |
| `attendance_employeebiometricenrollment` | ربط موظف ↔ رقم على الجهاز |
| `attendance_attendancepunch` | **سجلات البصمة الفعلية** |

## 10. فحص شامل لكل بيانات الإنتاج

```bash
cd /app
python manage.py check_production_data
python manage.py check_production_data --details
```

أو بعد النشر: `check-production --details`

يفحص: اتصال `DATABASE_URL` الفعلي، كل الجداول الأساسية، أعداد السجلات (موظفون، فروع، بصمات، مستخدمون…)، وسلامة الربط.

**قاعدة موحدة:** ضع **نفس** `DATABASE_URL` (Neon pooler) في Dokploy وفي `.env` المحلي. إن رأيت بصمات على الموقع وجداول فارغة في Neon، فالسيرفر ما زال على Postgres داخلي — راجع `docs/قاعدة-بيانات-موحدة.md`.
